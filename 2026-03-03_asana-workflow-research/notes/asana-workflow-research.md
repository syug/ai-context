# Asana Workflow / Rules 調査結果

## 背景

- Asana プロジェクト「AI Mission Control」(ID: `1213488615214853`) を運用中
- Claude Code がタスク作成・コメント追加・タスク完了を行う
- 現在、Asana Workflow で「タスク作成時」に Slack #claude-code-updates へ通知済み
- 追加トリガー（コメント追加時、タスク完了時）の実現可能性を調査

---

## 1. Asana Rules で利用可能なトリガー

Asana の「Rules」（旧称 Workflows）は、プロジェクト単位で設定する自動化機能。以下のトリガーが利用可能。

### タスク関連トリガー

| トリガー | 対応状況 | 備考 |
|---|---|---|
| タスクがプロジェクトに追加された | ✅ 対応 | 現在使用中。タスク作成時に発火 |
| タスクが完了になった | ✅ 対応 | `Task marked complete` トリガーあり |
| タスクが未完了に戻された | ✅ 対応 | |
| 担当者が変更された | ✅ 対応 | |
| 期日が変更された | ✅ 対応 | |
| 期日が近づいた | ✅ 対応 | |
| タスクが特定のセクションに移動した | ✅ 対応 | |
| タスクの添付ファイルが追加された | ✅ 対応 | |

### カスタムフィールド関連トリガー

| トリガー | 対応状況 | 備考 |
|---|---|---|
| カスタムフィールドが変更された | ✅ 対応 | 特定の値への変更も指定可能 |
| カスタムフィールドが特定の値に設定された | ✅ 対応 | |

### コメント関連トリガー

| トリガー | 対応状況 | 備考 |
|---|---|---|
| コメントが追加された | ❌ **非対応** | **Asana Rules にはコメント追加トリガーが存在しない** |

### その他

| トリガー | 対応状況 | 備考 |
|---|---|---|
| フォームが送信された | ✅ 対応 | |
| 承認ステータスが変更された | ✅ 対応 | |
| 外部Webhookからのトリガー | ✅ 対応 | API経由で Rule を発火可能 |

### 重要な制限事項

**コメント追加はトリガーとして使えない。** これは Asana コミュニティフォーラムでも長年要望されているが、2026年3月時点でも未実装。Asana Rules はタスクのフィールド変更やステータス変更に対応しているが、タスクへのコメント（ストーリー）追加にはトリガーが用意されていない。

## 2. Asana Rules で利用可能な Slack アクション

Asana + Slack 連携をインストールすると、Rules のアクション側で以下が使える。

| アクション | 説明 |
|---|---|
| **Send a Slack channel message** | 指定した Slack チャンネルにメッセージを投稿 |
| **Send a Slack DM** | 指定したユーザーに Slack DM を送信 |

### メッセージのカスタマイズ

- **変数の埋め込み可能:** タスク名、担当者名、期日、プロジェクト名、タスクURL などを変数として挿入できる
- **カスタムテキスト:** 固定テキストと変数を組み合わせたメッセージを作成可能
- **コメント内容の埋め込み:** コメントトリガーが存在しないため、コメント本文を変数として使うことはできない
- **投稿形式:** Asana bot 名義で投稿される（通常のテキストメッセージ）

### Slack アクションの具体的な設定例

```
トリガー: タスクが完了になった
アクション: Send a Slack channel message
チャンネル: #claude-code-updates
メッセージ: "✅ タスク完了: {Task Name} (担当: {Assignee})"
```

## 3. レート制限・頻度制限

### Asana Rules の制限

- **Rules の数:** Business プラン以上で無制限（Free/Premium は制限あり）
- **実行頻度:** 明示的なレート制限はドキュメント化されていないが、短時間に大量のルール発火が発生した場合、遅延や一時的なスキップが報告されている
- **同時発火:** 1つのタスク変更で複数ルールが発火する場合、順次実行される（並列ではない）
- **Amazon の Asana プラン:** Enterprise ライセンスのため、Rules 機能はフル利用可能

### 実用上の問題

Claude Code の使い方では、1セッションで数個のタスク操作（作成1回、コメント数回、完了1回）程度なので、レート制限は実質問題にならない。

## 4. メッセージカスタマイズの詳細

### 利用可能な変数（Slack メッセージ内で使用可能）

- `{Task Name}` - タスク名
- `{Assignee}` - 担当者
- `{Due Date}` - 期日
- `{Project Name}` - プロジェクト名
- `{Section}` - セクション
- `{Task URL}` - タスクへのリンク
- カスタムフィールドの値

### カスタマイズの限界

- HTML やリッチフォーマットは不可（プレーンテキストのみ）
- 条件分岐（if/else）はメッセージ内では不可
- コメント本文のテキストは変数として使えない

## 5. Rules vs Workflows（用語の整理）

Asana では以下のように用語が使われている。

| 用語 | 説明 |
|---|---|
| **Rules** | プロジェクト内の自動化ルール。トリガー + アクションの組み合わせ。最も一般的な自動化手段 |
| **Workflows** | Rules を含むより広い概念。Workflow Builder UI から Rules を作成・管理する |
| **Bundles** | 複数の Rules をまとめたもの。テンプレートとして共有可能 |

実質的に「Rules」と「Workflows」は同じ機能を指す。設定場所は:
- プロジェクト > Customize > Rules
- または、プロジェクト > Workflow Builder（ビジュアルエディタ）

## 6. コメント通知の代替アプローチ

コメント追加トリガーが Asana Rules に存在しないため、以下の代替案を検討する。

### 案A: Asana Slack 連携の「Activity Feed」機能

- Asana + Slack 連携で `/asana link` コマンドを使い、プロジェクトを Slack チャンネルにリンク
- プロジェクトの全アクティビティ（コメント含む）がチャンネルに自動投稿される
- **メリット:** 設定簡単、コメントも含むすべての変更が通知される
- **デメリット:** 通知が冗長になりやすい。フィルタリングが粗い（全アクティビティ or なし）

### 案B: Claude Code 側で直接 Slack に通知

- Claude Code がコメント追加時に、別途 Slack Incoming Webhook / Slack API を呼び出す
- **メリット:** 完全にカスタマイズ可能。メッセージ内容、フォーマット、条件を自由に制御
- **デメリット:** Slack Incoming Webhook の設定が必要。Amazon corporate Slack での Webhook 利用可否の確認が必要

### 案C: Asana API Webhooks + 中間サーバー

- Asana Events API / Webhooks で全イベントを受信し、中間サーバーでフィルタリングして Slack に転送
- **メリット:** 最も柔軟。コメント追加を含むあらゆるイベントを捕捉可能
- **デメリット:** サーバー運用が必要。オーバーエンジニアリングの可能性大

### 案D: タスクの「コメント追加」をカスタムフィールド更新で代替

- Claude Code がコメント追加時に、同時にカスタムフィールド（例: `Last Update`）を更新
- カスタムフィールド変更をトリガーにして Slack 通知
- **メリット:** Asana Rules の枠内で完結
- **デメリット:** Claude Code 側の実装変更が必要。カスタムフィールドが本来の用途と異なる

## 7. Slack Incoming Webhook との比較

### Amazon Corporate Slack での Webhook 利用

| 方式 | 利用可能性 | 特徴 |
|---|---|---|
| **Asana Rules → Slack** | ✅ 即利用可能 | Asana 公式連携。設定簡単だがトリガー制限あり |
| **Slack Incoming Webhook** | ⚠️ 要確認 | Amazon Slack では Incoming Webhook App がチーム/Workspace 管理者の承認が必要な場合あり |
| **Slack Workflow Builder** | ✅ 利用可能 | Slack 側のワークフローだが、外部トリガー（Webhook）受信が可能 |
| **Custom Slack App** | ⚠️ 要確認 | Amazon Slack App Directory での承認プロセスが必要 |

### 推奨アプローチ

**コメント通知が不要な場合（推奨パス）:**

1. **タスク作成** → Asana Rules で Slack 通知 ✅（実装済み）
2. **タスク完了** → Asana Rules で Slack 通知 ✅（追加設定のみ）
3. **コメント追加** → 通知なし（Asana タスクを見に行けばよい）

この方式が最もシンプルで保守コストが低い。

**コメント通知も必要な場合:**

優先度順:
1. **案A:** `/asana link` でプロジェクト全体を Slack チャンネルにリンク → 最も手軽
2. **案D:** Claude Code でコメント時にカスタムフィールドも更新 → 中程度の工数
3. **案B:** Slack Incoming Webhook を直接呼ぶ → 柔軟だが Webhook の利用可否確認が必要

## 8. 推奨アクションプラン

### 即座に実行可能（5分）

1. **タスク完了トリガーの追加**
   - Asana プロジェクト「AI Mission Control」を開く
   - Customize > Rules に移動
   - 新しい Rule 作成:
     - トリガー: `Task marked complete`
     - アクション: `Send a Slack channel message` to `#claude-code-updates`
     - メッセージ: `Task completed: {Task Name}`

### 検討事項

2. **コメント通知の要否を判断**
   - 現在の運用で、コメント通知がないことで困る場面があるか？
   - Claude Code のコメントは進捗報告・質問・エラー通知に使っているが、これらはスマホ通知が必須か？
   - Asana アプリ自体の通知（プッシュ通知）で代替可能ではないか？

3. **Asana モバイルアプリの通知設定確認**
   - Asana アプリはコメント追加時にプッシュ通知を送る
   - タスクのコラボレーターに入っていれば通知される
   - Slack 経由でなくても、Asana アプリで直接通知を受け取れる

## まとめ

| 通知イベント | Asana Rules 対応 | 推奨方式 |
|---|---|---|
| タスク作成 | ✅ | Asana Rules → Slack（実装済み）|
| タスク完了 | ✅ | Asana Rules → Slack（**追加設定推奨**）|
| コメント追加 | ❌ | Asana アプリ通知 or `/asana link` |
| ステータス変更 | ✅ | 必要に応じて Rules 追加可能 |
| カスタムフィールド変更 | ✅ | 必要に応じて Rules 追加可能 |

**結論:** タスク完了トリガーは即座に追加可能。コメント通知は Asana Rules では不可能なため、Asana アプリの通知機能を使うか、`/asana link` でプロジェクトを Slack にリンクするのが最も現実的。
